## Build System ŌĆö Ground Truth

**The working build uses Conan 2.x + MSYS2 ucrt64 GCC 15.2.**

| Item | Value |
| --- | --- |
| Compiler | `C:/msys64/ucrt64/bin/g++.exe` (GCC 15.2) |
| Make | `C:/msys64/ucrt64/bin/mingw32-make.exe` |
| Generator | MinGW Makefiles |
| Toolchain file | `build/conan_toolchain.cmake` (generated by `conan install`) |
| Conan | `C:/msys64/ucrt64/bin/conan.exe` (installed via MSYS2 pacman) |
| Conan profile | `~/.conan2/profiles/liteaero-gcc` |
| Dependency management | **Conan 2.x** for all deps; FetchContent for mcap (not in ConanCenter) |
| Eigen3, nlohmann_json, gtest, protobuf | All from Conan (`conanfile.txt`) |

**`conanfile.txt` is fully integrated into the build.** All C++ dependencies except mcap are declared there.

**Do not use any other `conan` binary** (e.g., from conda or system Python). The correct binary is `C:/msys64/ucrt64/bin/conan.exe`.

**ucrt64/bin must be on PATH** when invoking conan, cmake, and mingw32-make:

```bash
export PATH="/c/msys64/ucrt64/bin:$PATH"
```

**Configure and build:**

```bash
# Step 1: Install Conan packages (once per build_type; Release is the default):
PATH="/c/msys64/ucrt64/bin:$PATH" conan install . --output-folder=build --build=missing --profile=liteaero-gcc

# Step 2: Configure (toolchain file is auto-loaded via CMakePresets; or pass explicitly):
PATH="/c/msys64/ucrt64/bin:$PATH" cmake -B build -G "MinGW Makefiles" \
    -DCMAKE_TOOLCHAIN_FILE=build/conan_toolchain.cmake \
    -DCMAKE_BUILD_TYPE=Release

# Step 3: Build and test ŌĆö PATH required for ctest too; without it Windows
# finds the wrong libstdc++ DLL and every test crashes with a dialog box:
PATH="/c/msys64/ucrt64/bin:$PATH" mingw32-make -C build
PATH="/c/msys64/ucrt64/bin:$PATH" ctest --test-dir build --output-on-failure
```

**Debug builds** require a separate `conan install` pass with `--settings build_type=Debug`, because the Conan-generated CMakeDeps wraps include directories and library paths in `$<$<CONFIG:Release>:...>` generator expressions that are invisible to Debug builds.

## Claude ŌĆö Non-Negotiable Build System Rules

These rules exist because of concrete past failures. Do not deviate from them.

1. **Verify before claiming.** Before asserting that any tool, library, or integration is "in use" or "working", read `build/CMakeCache.txt`. The presence of a config file (e.g., `conanfile.txt`) does not mean it is integrated into the build. If `build/CMakeCache.txt` does not reference it, it is not in use.

2. **Identify tool binaries from the project's build, not by searching.** Before running cmake, make, conan, or any other build tool, identify the correct binary from `build/CMakeCache.txt` or ask the user. Never use a binary found by searching the filesystem that comes from an unrelated project or environment.

3. **Do not fabricate build system state.** If you are uncertain whether a dependency, tool, or integration is working, say so and ask. Do not invent a state and report it as fact.

## Quick Reference

### Non-Negotiable Rules

1. **No unsolicited implementation** ŌĆö Do not write, edit, or generate source code, test code, or build-system files unless the user has explicitly instructed you to implement or code something in the current message. Reading files, updating documentation, and updating implementation-plan documents are always permitted. When in doubt, ask rather than implement. Exception: empty files (no code, no design, no implementation) may be deleted without explicit instruction ŌĆö they do not embody past design effort.
2. **TDD** ŌĆö Write a failing test before writing production code.
3. **SI units** ŌĆö All stored values are meters, radians, seconds, kilograms, newtons. No exceptions inside the domain layer.
4. **Unit conversion** ŌĆö Only at the outermost interface (display, config file parsing). Never inside computation code.
5. **Serialization** ŌĆö Every stateful class implements both `serializeJson()`/`deserializeJson()` (nlohmann/json) and `serializeProto()`/`deserializeProto()` (protobuf v3). Round-trip tests are required for both formats.
6. **Naming** ŌĆö Names are self-documenting. Abbreviations and Hungarian notation are forbidden. Encode units in names when not obvious from context.
7. **Initial development phase ŌĆö no backward compatibility, no schema version iteration** ŌĆö The project is in initial development. There are no deployed APIs or serialized files to protect. Do not add forwarding shims, deprecated aliases, or compatibility wrappers. When an API changes, update all call sites directly and delete the old form. When code is removed, remove it completely. `schema_version` is always `1`; never increment it. When the project transitions to a maintenance phase this policy will change, but the versioning infrastructure is already in place for that transition.
8. **American English** ŌĆö Use American spellings in all comments, documentation, identifiers, and string literals (e.g. "color" not "colour", "serialize" not "serialise").
9. **Documentation reflects the present** ŌĆö Keep all documentation accurate to the current state of the code. Remove or update any section that describes a previous design or a state that no longer exists. Do not leave historical notes inline; version control preserves history. Proposed extensions to the architecture (new subsystems, planned classes, future algorithms) may be documented, but must be clearly labeled as proposed or not yet implemented ŌĆö never described as if they already exist.
10. **Surface cleanup issues as work items** ŌĆö When a cleanup issue is discovered (e.g. a misplaced include, a stale using-declaration, a redundant dependency), document the gap in the relevant implementation or architecture doc and add a work item to the project roadmap. Do not wait for the user to stumble across it. Fixing it is a separate task that requires an explicit instruction.
11. **Document references use relative-path hyperlinks** ŌĆö All references to other documents in documentation files (`.md`) must be expressed as Markdown hyperlinks with relative paths: `[display text](relative/path/to/file.md)`. Bare backtick paths like `` `docs/foo.md` `` without a link are not permitted for document references. Cross-repository references (e.g. `liteaero-sim/docs/...`) that cannot be linked relatively may remain as inline code.

## Key Conventions at a Glance

| Topic | Rule |
| --- | --- |
| C++ naming | `PascalCase` classes, `camelCase` methods, `snake_case_` members, `SCREAMING` constants |
| Python naming | `PascalCase` classes, `snake_case` functions/variables, `SCREAMING` constants |
| Booleans | Named as predicates: `is_converged`, `has_waypoint` |
| Units in names | `altitude_m`, `roll_rate_rad_s`, `thrust_n` when not obvious |
| Test naming | Mirror source file with `_test` suffix (C++) or `test_` prefix (Python) |
| Serialized fields | SI unit suffix in field name: `"altitude_m"`, `"roll_rate_rad_s"` |
| Schema version | Field `"schema_version"` (int) in every serialized object, always `1` |
| New dependency | Check license first; add to `conanfile.txt` if in ConanCenter; FetchContent otherwise |
| Angular velocity notation | $\boldsymbol{\omega}_{A/B}^C$: rotation of frame $A$ relative to frame $B$, expressed in frame $C$ |
| Aspect ratio in math | `$A\!\!R$` (double negative kern) ŌĆö not plain `$A$` |
| Subscript chaining | Use multi-step subscripting: $x_{LE_{HT}}$ ŌĆö not comma notation $x_{LE,HT}$ |
| Airframe abbreviations | HT = horizontal tail, VT = vertical tail, LE = leading edge, TE = trailing edge, QC = quarter chord |
